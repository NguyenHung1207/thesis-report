\section{Xây dựng các thành phần chính}

\subsection{Giao diện người dùng}

Giao diện người dùng được xây dựng bằng ReactJS, Vite, shadcn/ui và Tailwind CSS. Frontend hiện thực các màn hình thao tác chính của quy trình kiểm thử, từ tạo project, tạo test case, chạy test, xem lịch sử thực thi đến phân tích kết quả.

Các nhóm màn hình chính gồm màn hình xác thực, dashboard, project detail, test case detail, test data management, test suite, object repository, replay script và test run report. Trong quá trình hiện thực, các màn hình này được tách theo từng luồng sử dụng: quản lý project và website mục tiêu, sinh hoặc chỉnh sửa test case, cấu hình dữ liệu kiểm thử, khởi tạo test run, theo dõi timeline thực thi và xem evidence sau khi chạy.

Một số thành phần giao diện được tái sử dụng giữa nhiều màn hình, chẳng hạn form nhập prompt, bảng danh sách test case, panel cấu hình chạy, timeline step log, khung xem screenshot/evidence và khu vực hiển thị replay script. Các thông tin kỹ thuật như selector, assertion hoặc JSON của replay script chỉ được hiển thị ở những khu vực cần chỉnh sửa nâng cao để tránh làm luồng sử dụng chính trở nên phức tạp.

\subsection{Backend API}

Backend API được xây dựng bằng Node.js và Express.js. Thành phần này giữ vai trò trung tâm trong việc xử lý nghiệp vụ, quản lý dữ liệu và điều phối các yêu cầu chạy test sang agent-worker.

Các nhóm API chính gồm authentication, project, scan, test case, execution, replay script, test data, test suite, object repository và report. Khi người dùng yêu cầu chạy test, backend tạo bản ghi test run, chuẩn bị payload gồm thông tin test case, input params, runtime config, browser profile và callback URL, sau đó gửi sang agent-worker. Khi worker hoàn tất, backend nhận callback, cập nhật trạng thái test run, lưu evidence summary và trả kết quả để frontend hiển thị.

Backend cũng chịu trách nhiệm kiểm tra quyền truy cập project, validate dữ liệu đầu vào, chuẩn hóa trạng thái thực thi và đảm bảo các dữ liệu như replay script, evidence hoặc dataset chỉ được truy cập trong phạm vi hợp lệ.

\subsection{Agent-worker}

Agent-worker được xây dựng bằng Python và FastAPI. Đây là thành phần chịu trách nhiệm thực thi kiểm thử trên trình duyệt, bao gồm chạy test bằng Browser-use agent, chạy replay script bằng Playwright, ghi nhận evidence và callback kết quả về backend.

Agent-worker hỗ trợ hai chế độ thực thi chính:

\begin{itemize}
    \item \textbf{Goal-based agent}: worker nhận goal, prompt, plan snapshot và input params, sau đó tạo task cho Browser-use agent để chạy kiểm thử trên trình duyệt.
    \item \textbf{Replay script}: worker đọc script JSON đã lưu và dùng Playwright để chạy lại các bước như navigate, fill, click và assertion.
\end{itemize}

Trong quá trình chạy, worker ghi nhận action history, current URL, screenshot, log, trạng thái từng bước, verdict cuối cùng và error message nếu có. Sau khi chạy xong, worker callback kết quả về backend. Nếu phiên chạy bằng agent thành công, hệ thống có thể chuyển lịch sử hành động thành replay script để sử dụng cho các lần chạy sau.

Locator fallback được xử lý trong luồng agent execution. Khi agent không nhận diện được phần tử theo cách ban đầu, worker có thể hỗ trợ agent quan sát lại giao diện, sử dụng thông tin ngữ cảnh hoặc thông tin định vị phần tử đã lưu để thử cách nhận diện khác. Cơ chế này chỉ hỗ trợ một số thay đổi nhỏ của giao diện và không thay thế việc cập nhật lại test case khi luồng nghiệp vụ thay đổi lớn.

\subsection{Cơ sở dữ liệu}

Cơ sở dữ liệu sử dụng PostgreSQL 16 để lưu trữ dữ liệu nghiệp vụ và dữ liệu thực thi của hệ thống. Các nhóm dữ liệu chính gồm người dùng và project, test case và replay script, test run và evidence, report, test data, test suite và các bảng hỗ trợ vận hành.

Một số dữ liệu có cấu trúc linh hoạt như replay script, plan snapshot, variables schema, runtime config snapshot, selector collection và evidence summary được lưu dưới dạng JSON/JSONB. Cách lưu này giúp hệ thống dễ mở rộng khi cần bổ sung loại action, assertion, metadata hoặc evidence mới.

Trong quá trình hiện thực, nhóm không đưa toàn bộ schema vào phần thiết kế chính của Chương~\ref{Chapter4}, mà chỉ trình bày các bảng cốt lõi phục vụ vòng đời kiểm thử. Các bảng phụ trợ như cấu hình chạy, phiên đăng nhập, bảng phân loại hoặc lịch sử sinh dữ liệu được giữ trong schema vật lý nhưng không làm trọng tâm trình bày.

\section{Tích hợp các công nghệ lõi}

\subsection{Tích hợp Browser-use}

Browser-use được tích hợp trong agent-worker để hỗ trợ chế độ goal-based agent. Khi nhận yêu cầu chạy test, worker xây dựng task text từ goal, prompt, plan snapshot, input params, base URL và allowed domains. Task này được truyền cho Browser-use agent để agent quan sát giao diện, suy luận hành động và thao tác trên website mục tiêu.

Quá trình chạy bằng Browser-use được tổ chức theo vòng lặp quan sát -- suy luận -- hành động. Sau mỗi bước, worker ghi nhận log, screenshot và action history để phục vụ báo cáo và sinh replay script. Khi agent gặp khó khăn trong việc nhận diện phần tử, hệ thống có thể cung cấp thêm ngữ cảnh giao diện hoặc thông tin định vị phần tử đã lưu để hỗ trợ agent thử cách nhận diện khác.

\subsection{Tích hợp Playwright}

Playwright được sử dụng làm lớp điều khiển trình duyệt trong hệ thống. Trong chế độ agent, Playwright hỗ trợ mở trình duyệt và thực hiện các hành động do agent yêu cầu. Trong chế độ replay, Playwright trực tiếp chạy script JSON đã lưu.

Các thao tác chính gồm mở website, tìm phần tử, thực hiện click/fill/navigate, kiểm tra assertion, chụp screenshot và ghi nhận thông tin phục vụ evidence. Việc sử dụng Playwright giúp các lần chạy replay có tính xác định cao hơn vì không cần LLM suy luận lại toàn bộ luồng kiểm thử.

Trong chế độ replay, hệ thống ưu tiên tính xác định. Nếu locator, action hoặc assertion trong script không còn phù hợp, test run được ghi nhận là failed hoặc error và evidence được lưu lại để người dùng phân tích. Hệ thống không gọi LLM để tự sửa toàn bộ kịch bản replay trong lúc chạy.

\subsection{Tích hợp Gemini Flash}

Gemini Flash được sử dụng trong các tác vụ liên quan đến ngôn ngữ tự nhiên và phân tích ngữ cảnh. Trong hệ thống, mô hình hỗ trợ phân tích prompt, sinh test case, hỗ trợ agent suy luận hành động và gợi ý nguyên nhân khi test thất bại.

Tuy nhiên, hệ thống không phụ thuộc hoàn toàn vào LLM trong mọi lần chạy. Sau khi agent chạy thành công và replay script được tạo, các lần kiểm thử hồi quy có thể dùng Playwright để chạy lại script mà không cần gọi LLM ở từng bước. Cách kết hợp này giúp giảm chi phí vận hành và tăng tính xác định của quá trình kiểm thử.

\section{Xây dựng các chức năng kiểm thử}

\subsection{Scan/crawl website}

Chức năng scan/crawl website được xây dựng để ghi nhận thông tin nền ban đầu về website trong phạm vi project. Quá trình scan bắt đầu từ base URL và duyệt qua các trang có thể truy cập trong phạm vi cấu hình. Kết quả scan được lưu dưới dạng sitemap, interaction map, trạng thái scan, thời điểm chạy và thông tin lỗi nếu có.

Thông tin scan không thay thế khả năng quan sát của agent, mà đóng vai trò cung cấp ngữ cảnh ban đầu cho việc tạo test case và thực thi kiểm thử.

\subsection{Sinh test case}

Chức năng sinh test case cho phép người dùng nhập prompt hoặc goal bằng ngôn ngữ tự nhiên. Hệ thống sử dụng AI để phân tích nội dung này và tạo ra test case có cấu trúc gồm tiêu đề, mục tiêu kiểm thử, test plan và các thông tin liên quan.

Sau khi test case được sinh, người dùng có thể xem lại, chỉnh sửa hoặc lưu vào project. Test case được lưu cùng prompt gốc, execution mode, plan snapshot, variables schema và AI model nếu có để hỗ trợ truy vết và chỉnh sửa sau này.

\subsection{Chạy test bằng agent}

Chức năng chạy test bằng agent là phần thực thi linh hoạt của hệ thống. Người dùng chọn một test case và yêu cầu chạy bằng agent. Backend tạo test run, gửi thông tin test case và cấu hình chạy sang agent-worker. Worker khởi tạo trình duyệt, truyền mục tiêu kiểm thử cho Browser-use agent và sử dụng Playwright để thao tác trên website.

Trong quá trình chạy, agent quan sát giao diện, gọi LLM để suy luận hành động tiếp theo và thực hiện các thao tác như mở trang, nhập dữ liệu, nhấn nút, điều hướng hoặc kiểm tra trạng thái. Nếu gặp vấn đề về nhận diện phần tử, agent có thể quan sát lại giao diện hoặc sử dụng thông tin phần tử đã lưu để thử cách nhận diện khác. Sau khi hoàn tất, hệ thống lưu log, screenshot, action history, evidence và verdict của test run.

\subsection{Replay script}

Replay script là kịch bản phát lại được sinh từ lịch sử hành động sau khi agent chạy thành công. Script được lưu dưới dạng JSON, bao gồm danh sách step, action type, params, assertion và metadata liên quan.

Khi chạy replay, worker sử dụng Playwright để thực thi từng step theo thứ tự. Replay script không dùng LLM để suy luận lại toàn bộ luồng nên phù hợp với kiểm thử hồi quy. Nếu script không còn phù hợp với giao diện hiện tại, hệ thống ghi nhận lỗi, lưu evidence và để người dùng cập nhật lại test case, replay script hoặc thông tin định vị phần tử khi cần.

\begin{lstlisting}[caption={Ví dụ cấu trúc replay script rút gọn}, label={lst:replay_script_example}]
{
  "steps": [
    {
      "action": "navigate",
      "url": "https://example.com/login"
    },
    {
      "action": "fill",
      "target": "username",
      "value": "{{username}}"
    },
    {
      "action": "click",
      "target": "login_button"
    },
    {
      "action": "assert_visible",
      "target": "dashboard_title"
    }
  ],
  "metadata": {
    "source": "agent_run",
    "version": 1
  }
}
\end{lstlisting}

Replay script được thiết kế dưới dạng JSON để dễ lưu trữ, chỉnh sửa và mở rộng. Mỗi step gồm loại hành động, đối tượng thao tác, tham số đầu vào và điều kiện kiểm tra nếu có. Các biến như \texttt{\{\{username\}\}} hoặc \texttt{\{\{password\}\}} có thể được thay thế bằng dữ liệu từ dataset khi chạy cùng một kịch bản với nhiều bộ dữ liệu.

\subsection{Quản lý test data và test suite}

Chức năng test data cho phép người dùng tạo và quản lý các bộ dữ liệu đầu vào cho test case. Dataset có thể được tạo thủ công hoặc sinh tự động bằng AI dựa trên mục tiêu kiểm thử. Khi chạy test với dataset, hệ thống sử dụng dữ liệu tương ứng làm input và lưu kết quả riêng cho từng lần chạy.

Test suite được xây dựng để gom nhóm nhiều test case và chạy theo lô. Người dùng có thể tạo suite, thêm test case vào suite, sắp xếp thứ tự chạy và khởi tạo suite run. Sau khi chạy, hệ thống tổng hợp kết quả từng test case trong suite để phục vụ kiểm thử hồi quy.

\subsection{Report và evidence}

Sau mỗi lần chạy test, hệ thống tổng hợp kết quả thành report. Report bao gồm trạng thái cuối cùng, verdict, thời gian chạy, danh sách bước đã thực hiện, screenshot, log, evidence và error message nếu có.

Evidence được ghi nhận để hỗ trợ truy vết và phân tích lỗi. Với mỗi test run, hệ thống có thể lưu screenshot, action history, step log, URL hiện tại hoặc artifact khác tùy theo cấu hình. Khi test thất bại, các evidence này giúp người dùng xác định bước xảy ra lỗi và đưa ra hướng sửa phù hợp.

Trong phần hiện thực, backend lưu thông tin evidence như loại evidence, test run liên quan, step liên quan, dữ liệu file hoặc metadata artifact, thời điểm tạo và mô tả ngắn. Các evidence như screenshot, log hoặc action history được gắn với từng test run để người dùng có thể mở lại từ report mà không cần truy cập trực tiếp vào worker.

\section{Xây dựng cơ chế hỗ trợ}

\subsection{Object repository và locator fallback}

Object repository được xây dựng để lưu trữ thông tin định vị phần tử giao diện theo project. Mỗi object đại diện cho một phần tử có ý nghĩa trong kiểm thử, chẳng hạn như ô nhập username, ô nhập password, nút login hoặc một trường dữ liệu trong form.

Thay vì chỉ lưu một selector duy nhất, hệ thống có thể lưu nhiều thuộc tính định vị khác nhau như test id, id ổn định, name attribute, CSS selector, accessible name, placeholder, title hoặc XPath. Các thông tin này cung cấp thêm ngữ cảnh để agent nhận diện phần tử khi giao diện thay đổi nhỏ.

Locator fallback trong phạm vi hệ thống được hiểu là cơ chế hỗ trợ trong quá trình agent thực thi. Khi cách nhận diện ban đầu không còn phù hợp, agent có thể quan sát lại giao diện và thử cách định vị khác dựa trên ngữ cảnh hoặc thông tin đã lưu. Cơ chế này không nhằm tự động sửa mọi thay đổi giao diện và không thay thế việc cập nhật lại test case khi website thay đổi lớn về luồng nghiệp vụ.

\subsection{AI analysis và report}

Khi test thất bại, hệ thống có thể sử dụng AI để hỗ trợ gợi ý nguyên nhân lỗi, chẳng hạn do website mục tiêu không thỏa điều kiện kiểm tra, locator không tìm thấy phần tử, timeout, dữ liệu đầu vào không phù hợp hoặc agent chọn sai hành động. Kết quả phân tích này chỉ đóng vai trò hỗ trợ, không thay thế việc kiểm tra lại của người dùng.

Thông tin đưa vào quá trình phân tích có thể gồm error message, action history, screenshot metadata, URL cuối cùng, trạng thái step và verdict. Dựa trên đó, hệ thống hiển thị gợi ý để người dùng cập nhật test case, dữ liệu kiểm thử, replay script hoặc thông tin định vị phần tử.

\section{Bảo mật và xử lý dữ liệu nhạy cảm}

Trong quá trình xây dựng hệ thống, một số nhóm dữ liệu cần được xem là nhạy cảm, bao gồm tài khoản người dùng, mật khẩu, API key, thông tin xác thực website, dữ liệu test account, screenshot và log. Do đó, hệ thống cần áp dụng các nguyên tắc bảo vệ dữ liệu trong quá trình hiện thực.

Các cơ chế chính gồm:

\begin{itemize}
    \item Không lưu mật khẩu người dùng ở dạng thô.
    \item Không để lộ API key hoặc cấu hình LLM ở frontend.
    \item Kiểm tra quyền truy cập project trước khi cho phép xem hoặc chỉnh sửa test case, dataset, replay script và evidence.
    \item Giới hạn domain được phép truy cập khi chạy agent hoặc replay script.
    \item Không ghi token, API key hoặc mật khẩu thật vào log nếu không cần thiết.
    \item Callback từ agent-worker về backend cần kèm callback secret để backend xác thực nguồn gửi trước khi cập nhật test run, step log hoặc evidence.
    \item Xem nội dung website là dữ liệu không tin cậy để hạn chế rủi ro prompt injection.
    \item Khuyến khích sử dụng tài khoản test hoặc tài khoản staging khi chạy kiểm thử.
\end{itemize}

Do hệ thống có khả năng tự động thao tác trên website, việc giới hạn phạm vi kiểm thử là đặc biệt quan trọng. Mỗi project cần xác định base URL và allowed domains để hạn chế trường hợp agent hoặc replay script điều hướng ra ngoài phạm vi mong muốn.

\section{Triển khai và tích hợp vận hành}

Hệ thống được triển khai theo mô hình tách thành nhiều service. Frontend được triển khai độc lập trên Vercel để phục vụ giao diện người dùng. Backend API và agent-worker được đóng gói bằng Docker và triển khai trên Azure. Cách triển khai này phù hợp với kiến trúc đã thiết kế, trong đó backend chịu trách nhiệm xử lý nghiệp vụ và điều phối, còn agent-worker xử lý các tác vụ nặng như khởi tạo trình duyệt, chạy agent, chạy replay script và ghi nhận evidence.

Việc tách backend API và agent-worker thành các service riêng giúp quá trình triển khai linh hoạt hơn. Agent-worker có thể được cấu hình tài nguyên cao hơn để phục vụ browser automation mà không ảnh hưởng trực tiếp đến backend nghiệp vụ. Các thành phần giao tiếp với nhau thông qua HTTP API và callback. Backend gửi yêu cầu thực thi sang worker, sau đó worker gửi kết quả từng bước và kết quả cuối cùng về backend để cập nhật test run.

Quy trình triển khai backend và agent-worker được tự động hóa bằng CI/CD. Khi có thay đổi mã nguồn trên nhánh triển khai tương ứng, pipeline sẽ đăng nhập vào Azure, build Docker image, đẩy image lên container registry và cập nhật service đang chạy trên Azure. Cách làm này giúp giảm thao tác thủ công khi cập nhật hệ thống, đồng thời đảm bảo mỗi lần triển khai sử dụng image mới nhất từ mã nguồn.

Frontend được triển khai trên Vercel và kết nối đến backend API thông qua URL cấu hình môi trường. Việc triển khai frontend độc lập giúp giao diện có thể được cập nhật nhanh chóng mà không cần build lại backend hoặc agent-worker. Đồng thời, các biến môi trường như URL backend, Google OAuth client ID hoặc các cấu hình frontend khác được quản lý qua môi trường triển khai thay vì ghi trực tiếp trong mã nguồn.

Các biến môi trường quan trọng của hệ thống gồm cấu hình database, JWT secret, LLM API key, địa chỉ agent-worker, địa chỉ backend callback, callback secret, cấu hình allowed domains và các URL frontend/backend tương ứng. Những thông tin này không được đặt trực tiếp trong frontend hoặc mã nguồn public. Đối với các service backend và agent-worker, các giá trị nhạy cảm được cấu hình thông qua secret hoặc environment variables của môi trường triển khai.

Nhờ mô hình triển khai này, hệ thống có thể vận hành theo luồng thực tế: người dùng thao tác trên frontend Vercel, frontend gọi backend API, backend lưu dữ liệu và gửi yêu cầu sang agent-worker trên Azure, agent-worker chạy kiểm thử trên trình duyệt rồi callback kết quả về backend để hiển thị lại trên giao diện người dùng.

\section{Khó khăn trong quá trình xây dựng}

Trong quá trình xây dựng hệ thống, nhóm gặp một số khó khăn chính sau:

\begin{itemize}
    \item \textbf{Tích hợp nhiều công nghệ}: hệ thống kết hợp ReactJS, Node.js, PostgreSQL, Python FastAPI, Browser-use, Playwright và LLM, nên cần cơ chế giao tiếp rõ ràng giữa các thành phần.
    \item \textbf{Quản lý trạng thái chạy test}: một test run có thể kéo dài, timeout hoặc phát sinh lỗi, nên cần lưu trạng thái và callback kết quả chính xác.
    \item \textbf{Chuyển lịch sử agent thành replay script}: hành động do agent thực hiện cần được chuẩn hóa thành các step có cấu trúc để có thể chạy lại bằng Playwright.
    \item \textbf{Xử lý locator không ổn định}: giao diện website có thể thay đổi, khiến selector cũ không còn hoạt động, nên cần object repository và locator fallback ở mức agent run.
    \item \textbf{Kiểm soát chi phí khi dùng LLM}: replay script được dùng để giảm số lần gọi LLM trong kiểm thử hồi quy.
    \item \textbf{Ghi nhận evidence hợp lý}: screenshot, log và action history cần đủ để phân tích lỗi nhưng không nên làm hệ thống quá tải.
\end{itemize}

Những khó khăn trên ảnh hưởng trực tiếp đến cách hiện thực hệ thống. Vì vậy, nhóm ưu tiên tách thành phần, lưu vết đầy đủ, dùng replay script để giảm chi phí LLM và bổ sung object repository để hỗ trợ bảo trì kịch bản kiểm thử.
