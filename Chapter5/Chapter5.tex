\paragraph{Giao diện Object Repository.}

Module Objects được dùng để quản lý các phần tử giao diện có thể tái sử dụng trong quá trình chạy replay script. Trong giao diện project, nhóm Objects xuất hiện ở sidebar và được gom theo Page/Group. Khi người dùng mở rộng một nhóm và chọn một object, màn hình chính hiển thị thông tin chi tiết của object đó như tên object, trạng thái, selector mặc định, các selector dự phòng, DOM properties snapshot và thông tin lần ghi nhận gần nhất.

Hình~\ref{fig:chapter5_object_repository_ui} minh họa giao diện Object Repository trong hệ thống. Ở phần đầu màn hình, người dùng có thể tạo object mới bằng nút \textit{New Object}. Với mỗi object đang được chọn, hệ thống hiển thị trạng thái \textit{Auto} hoặc \textit{Confirmed}. Trạng thái \textit{Auto} cho biết object được hệ thống tự ghi nhận từ phiên chạy agent, trong khi trạng thái \textit{Confirmed} cho biết người dùng đã xác nhận object này để sử dụng ổn định hơn trong các lần replay sau.

Giao diện chi tiết object cho phép người dùng xem và chỉnh sửa các thông tin chính gồm Object Name, Page/Group, Description, Default Selector và Selector Collection. Default Selector là locator được ưu tiên sử dụng trước, còn Selector Collection lưu nhiều cách định vị khác nhau như CSS, XPath, \texttt{id}, \texttt{name}, placeholder, accessible name, text hoặc \texttt{data-testid}. Người dùng có thể thêm fallback locator, sao chép locator, chỉnh sửa object hoặc xóa object khi không còn sử dụng.

Bên cạnh các locator đã lưu, giao diện còn hiển thị DOM properties snapshot và Candidate Locators. Candidate Locators là các locator mới được phát hiện trong những lần chạy sau. Đối với object đã được người dùng xác nhận, hệ thống không tự động ghi đè locator hiện có mà đưa locator mới vào danh sách candidate để người dùng chọn \textit{Accept} hoặc \textit{Dismiss}. Cách thiết kế này giúp giảm rủi ro làm hỏng replay script do cập nhật locator ngoài ý muốn.

\subsection{Agent-worker}

Agent-worker được xây dựng bằng Python và FastAPI, chịu trách nhiệm trực tiếp chạy kiểm thử trên trình duyệt. Thành phần này hỗ trợ hai chế độ:

\begin{itemize}
    \item \textbf{Goal-based agent}: nhận goal, test plan, dữ liệu đầu vào và cấu hình runtime để Browser-use agent thực thi kiểm thử.
    
    \item \textbf{Replay script}: đọc script JSON và sử dụng Playwright để phát lại các step.
\end{itemize}

Trong quá trình chạy, worker ghi nhận action history, URL, screenshot, log, trạng thái step, verdict và error message. Sau khi hoàn tất, worker callback kết quả về backend; phiên agent thành công có thể được chuyển thành replay script.

\begin{table}[H]
\centering
\caption{Nhóm endpoint và callback chính của agent-worker}
\label{tab:chapter5_worker_endpoints}
\renewcommand{\arraystretch}{1.2}
\begin{tabular}{|p{4.0cm}|p{10.2cm}|}
\hline
\textbf{Nhóm giao tiếp} &
\textbf{Vai trò hiện thực} \\
\hline

Health check &
Kiểm tra trạng thái sẵn sàng của worker. \\
\hline

Run agent &
Nhận goal, test plan, dữ liệu đầu vào, base URL, allowed domains và callback URL. \\
\hline

Run replay script &
Nhận replay script và phát lại các step bằng Playwright. \\
\hline

Step callback &
Gửi trạng thái step, URL, log hoặc thông tin lỗi về backend. \\
\hline

Evidence callback &

\hline

Final callback &
Gửi verdict, trạng thái cuối cùng, thời gian chạy và error message. \\
\hline
\end{tabular}
\end{table}

Locator fallback được xử lý trong luồng agent khi cách nhận diện phần tử ban đầu không còn phù hợp. Cơ chế này chỉ hỗ trợ các thay đổi nhỏ và không thay thế việc cập nhật lại test case khi luồng nghiệp vụ thay đổi.

% ---------------------------------------------------------
\subsection{Cơ sở dữ liệu}

PostgreSQL 16 được sử dụng để lưu dữ liệu nghiệp vụ và dữ liệu thực thi. Các nhóm dữ liệu chính gồm người dùng, project, test case, replay script, test run, evidence, dataset, test suite và report.

Replay script, test plan snapshot, variables schema, runtime config, selector collection và evidence summary được lưu bằng JSON/JSONB khi cần cấu trúc linh hoạt. Các bảng cốt lõi đã được trình bày ở Chương~\ref{Chapter4}; Chương này tập trung vào cách dữ liệu được sử dụng trong quá trình hiện thực.

\subsection{Tích hợp Browser-use}

Browser-use được tích hợp trong agent-worker để thực hiện goal-based agent. Worker tạo task từ goal, test plan, dữ liệu đầu vào, base URL và allowed domains; agent sau đó quan sát giao diện, suy luận và thực hiện hành động.

Sau mỗi bước, worker ghi nhận action history, screenshot và log để phục vụ report và tạo replay script.

% ---------------------------------------------------------
\subsection{Tích hợp Playwright}

Playwright được dùng để điều khiển trình duyệt và thực thi các action như navigate, click, fill, assertion và screenshot. Trong chế độ replay, Playwright chạy trực tiếp script JSON theo thứ tự mà không cần LLM suy luận lại ở từng bước.

Nếu locator, action hoặc assertion không còn phù hợp, hệ thống ghi nhận \texttt{failed} hoặc \texttt{error} và lưu evidence để phân tích.

% ---------------------------------------------------------
\subsection{Tích hợp Gemini Flash}

Gemini Flash hỗ trợ phân tích prompt, sinh test case, sinh dữ liệu, suy luận hành động cho agent và phân tích kết quả thực thi.

Sau khi replay script được tạo, các lần kiểm thử hồi quy có thể chạy bằng Playwright mà không cần gọi LLM ở từng step, qua đó giảm chi phí và tăng tính lặp lại.

% =========================================================
\section{Xây dựng các chức năng kiểm thử}
% =========================================================

% ---------------------------------------------------------
\subsection{Scan/crawl website}

Chức năng scan/crawl bắt đầu từ base URL và thu thập các URL, sitemap, interaction map, trạng thái và thông tin lỗi trong phạm vi project. Kết quả scan cung cấp ngữ cảnh ban đầu cho quá trình sinh test case và chạy agent.

% ---------------------------------------------------------
\subsection{Sinh test case}

Người dùng nhập prompt hoặc goal bằng ngôn ngữ tự nhiên. AI tạo test case gồm tiêu đề, mục tiêu, Test Plan, dữ liệu đầu vào và kết quả kỳ vọng.

Test case ứng viên được lưu tạm để người dùng xem lại; các ứng viên được chọn mới được chuyển vào danh sách test case chính thức của project.

% ---------------------------------------------------------
\subsection{Chạy test bằng agent}

Backend tạo test run và gửi test case cùng cấu hình sang agent-worker. Worker khởi tạo trình duyệt, sử dụng Browser-use và Gemini Flash để quan sát giao diện, suy luận hành động và thực thi bằng Playwright.

Trong quá trình chạy, hệ thống lưu step log, screenshot, action history, evidence và verdict. Khi thành công, lịch sử hành động có thể được chuẩn hóa thành replay script.

% ---------------------------------------------------------
\subsection{Replay script}

Replay script được sinh từ lịch sử hành động của một phiên agent thành công và được lưu dưới dạng JSON. Script gồm các step, action, target, tham số, check hoặc assertion và metadata liên quan.

\begin{lstlisting}[
    caption={Ví dụ cấu trúc replay script rút gọn},
    label={lst:replay_script_example}
]
{
  "steps": [
    {
      "action": "navigate",
      "url": "https://example.com/login"
    },
    {
      "action": "fill",
      "target": "username",
      "value": "{{username}}"
    },
    {
      "action": "click",
      "target": "login_button"
    },
    {
      "action": "assert_visible",
      "target": "dashboard_title"
    }
  ],
  "metadata": {
    "source": "agent_run",
    "version": 1
  }
}
\end{lstlisting}

Playwright thực thi các step theo thứ tự. Các biến như \texttt{\{\{username\}\}} hoặc \texttt{\{\{password\}\}} được thay thế bằng dữ liệu của từng dòng trong dataset.

Các check hoặc assertion trong script được sử dụng để xác định trạng thái cuối cùng của test run. Người dùng có thể chỉnh sửa chúng trong Developer Mode.

% ---------------------------------------------------------
\subsection{Quản lý dữ liệu kiểm thử và test suite}

Dataset có thể được tạo thủ công tại module Data hoặc sinh bằng AI từ một test case. Dữ liệu được lưu theo dạng bảng và có thể được chỉnh sửa, xem trước hoặc xóa.

Khi chạy replay script, hệ thống kiểm tra số lượng trường và kiểu dữ liệu của dataset. Dataset không tương thích hoặc cấu hình chưa đầy đủ sẽ bị chặn; dataset sinh trực tiếp từ test case được tạo theo schema đầu vào tương ứng.

Test suite gom nhiều test case thành một cấu hình chạy theo lô. Mỗi test case có thể chạy bằng agent hoặc replay script; replay script có thể kết hợp với dataset để tạo nhiều lượt thực thi.

Kết quả suite được tổng hợp theo số lượt thực thi, số lượt pass, fail, tỷ lệ pass và tổng thời gian. Người dùng có thể mở từng kết quả để xem chi tiết.

% ---------------------------------------------------------
\subsection{Report và evidence}

Kết quả được tổ chức thành test case run, suite run và dataset run. Mỗi report lưu trạng thái cuối cùng, thời gian, step, action, URL, screenshot, error message và failure type.

Suite run bổ sung kết quả tổng hợp của các lượt thực thi. Dataset run lưu dữ liệu đầu vào và kết quả riêng cho từng dòng dữ liệu.

Evidence gồm screenshot, step log, action history, URL và các artifact liên quan. Screenshot được gắn với step tương ứng; log kỹ thuật được lưu ở backend để hỗ trợ truy vết và AI Analysis nhưng không hiển thị trực tiếp trên giao diện chính.

\section{Xây dựng cơ chế hỗ trợ}
% =========================================================

% ---------------------------------------------------------
\subsection{Object Repository và locator fallback}

Object Repository được hiện thực để lưu các phần tử giao diện được phát hiện trong quá trình chạy agent và cho phép tái sử dụng trong replay script. Khi một phiên chạy agent kết thúc, agent-worker trích xuất các object từ lịch sử thao tác đã được chuẩn hóa, bao gồm tên object, Page/Group, selector method, selector collection, URL nguồn và một số thuộc tính DOM. Các object này được gửi về backend cùng kết quả cuối cùng của test run.

Ở backend, object mới được lưu vào bảng \texttt{test\_objects} với trạng thái \textit{Auto}. Nếu object đã tồn tại và vẫn ở trạng thái \textit{Auto}, hệ thống có thể cập nhật locator mới nhất cho object đó. Nếu object đã được người dùng xác nhận, hệ thống không ghi đè trực tiếp mà lưu locator mới vào \texttt{test\_object\_candidates}. Người dùng có thể xem các candidate này trên giao diện Object Repository và quyết định chấp nhận hoặc bỏ qua.

Khi chạy replay script, các step có thể tham chiếu object thông qua \texttt{objectRef}. Trước khi gửi yêu cầu sang agent-worker, backend lấy selector mới nhất từ Object Repository và đưa vào step dưới dạng selector mặc định và selector collection. Ở worker, replay script thử selector mặc định trước, sau đó lần lượt thử các selector dự phòng theo thứ tự ưu tiên như \texttt{data-testid}, \texttt{id}, \texttt{name}, CSS, accessible name, placeholder, title và XPath. Cơ chế này hỗ trợ locator fallback trong một số trường hợp giao diện thay đổi nhỏ, nhưng không thay thế việc người dùng cập nhật lại test case hoặc replay script khi website thay đổi mạnh về luồng nghiệp vụ.

% ---------------------------------------------------------
\subsection{AI Analysis}

Khi người dùng yêu cầu phân tích, backend tập hợp error message, failure type, action history, trạng thái step, URL, screenshot metadata, log kỹ thuật và verdict để gửi cho Gemini Flash.

AI Analysis xác định bước xảy ra lỗi, mô tả nguyên nhân và đề xuất hướng xử lý. Với suite run, hệ thống có thể tổng hợp kết quả của nhiều lượt thực thi.

Kết quả phân tích được lưu cùng phiên chạy và hiển thị lại khi người dùng mở report. Nội dung này có vai trò hỗ trợ và không thay thế việc kiểm tra của người dùng.

\section{Bảo mật và xử lý dữ liệu nhạy cảm}
% =========================================================

Các dữ liệu nhạy cảm của hệ thống gồm mật khẩu, API key, token, tài khoản kiểm thử, dataset, screenshot và log. Các biện pháp chính được áp dụng gồm:

\begin{itemize}
    \item Không lưu mật khẩu người dùng ở dạng thô.
    
    \item Không để API key, JWT secret hoặc cấu hình LLM ở frontend.
    
    \item Kiểm tra quyền truy cập project trước khi truy cập test case, dataset, replay script và evidence.
    
    \item Giới hạn domain được phép truy cập khi chạy agent hoặc replay script.

    \item Xem nội dung website mục tiêu là dữ liệu không tin cậy để hạn chế rủi ro prompt injection~\cite{11359704}.
    
    \item Hạn chế ghi token, API key và mật khẩu vào log.
    
    \item Quản lý screenshot, dataset và evidence theo phạm vi người dùng và project.
    
    \item Khuyến khích sử dụng tài khoản test hoặc tài khoản staging khi thực thi kiểm thử.
\end{itemize}

Các giá trị nhạy cảm được cấu hình thông qua biến môi trường hoặc secret của nền tảng triển khai thay vì ghi trực tiếp trong mã nguồn.

\section{Khó khăn trong quá trình xây dựng}
% =========================================================

Trong quá trình xây dựng, nhóm gặp các khó khăn chính sau:

\begin{itemize}
    \item \textbf{Tích hợp nhiều công nghệ}: cần thống nhất cơ chế giao tiếp giữa ReactJS, Node.js, PostgreSQL, FastAPI, Browser-use, Playwright và LLM.
    
    \item \textbf{Quản lý trạng thái thực thi}: test run có thể kéo dài, timeout hoặc lỗi nên cần callback và lưu trạng thái chính xác.
    
    \item \textbf{Tạo replay script}: action history của agent phải được chuẩn hóa thành các step có thể phát lại.
    
    \item \textbf{Locator không ổn định}: thay đổi giao diện có thể khiến selector cũ không còn phù hợp.
    
    \item \textbf{Chi phí LLM}: replay script được sử dụng để giảm số lần gọi mô hình trong kiểm thử hồi quy.
    
    \item \textbf{Quản lý evidence}: screenshot, log và action history phải đủ để phân tích nhưng không làm tăng quá nhiều chi phí lưu trữ.
\end{itemize}

Các khó khăn trên dẫn đến việc hệ thống được tách thành nhiều service, lưu vết theo từng test run và kết hợp agent với replay script.
