\section{Kiến trúc tổng thể hệ thống}
% =========================================================

Từ các yêu cầu đã phân tích ở Chương~\ref{Chapter3}, hệ thống được thiết kế theo mô hình nhiều thành phần, gồm giao diện người dùng, backend API, cơ sở dữ liệu và agent-worker. Kiến trúc này hỗ trợ toàn bộ vòng đời kiểm thử: quản lý project, sinh test case, chạy kiểm thử bằng goal-based agent (sau đây gọi tắt là agent), tạo replay script, chạy kiểm thử hồi quy và phân tích kết quả.

Các mục tiêu thiết kế chính gồm:

\begin{itemize}
    \item \textbf{Tách biệt trách nhiệm}: frontend xử lý tương tác; backend quản lý nghiệp vụ; agent-worker thực thi kiểm thử trên trình duyệt.
    
    \item \textbf{Hỗ trợ ngôn ngữ tự nhiên}: người dùng có thể mô tả mục tiêu kiểm thử bằng prompt hoặc goal.
    
    \item \textbf{Kết hợp agent và replay script}: agent phù hợp với lần chạy linh hoạt ban đầu; replay script phù hợp với kiểm thử hồi quy.
    
    \item \textbf{Lưu vết thực thi}: mỗi test run cần lưu trạng thái, verdict, step log, screenshot, evidence và thông tin lỗi.
    
    \item \textbf{Hỗ trợ bảo trì}: Object Repository, assertion và locator fallback cung cấp thêm thông tin khi giao diện website thay đổi.
    
    \item \textbf{Kiểm soát chi phí và tài nguyên}: giảm số lần gọi LLM khi đã có replay script và giới hạn số phiên trình duyệt đồng thời.
\end{itemize}

\subsection{Thiết kế giao diện người dùng}

Giao diện được thiết kế theo vòng đời kiểm thử: xác thực người dùng, quản lý project, sinh test case, chuẩn bị dữ liệu, cấu hình thực thi và xem kết quả. Các màn hình chính gồm đăng nhập, đăng ký, Dashboard, Test Case Detail, Test Suite Detail, Data, Objects và Test Runs.

Thiết kế ưu tiên người dùng không phải viết script ngay từ đầu. Các chức năng nhập prompt, chạy bằng agent, sử dụng replay script, chọn dataset và xem evidence được tổ chức theo một luồng thao tác thống nhất.

% ---------------------------------------------------------
\subsection{Thiết kế backend API}

Backend API được tổ chức theo các nhóm tài nguyên như authentication, projects, test cases, datasets, test suites, test runs, execution scripts, evidences và reports.

Backend chịu trách nhiệm:

\begin{itemize}
    \item Xử lý đăng ký, đăng nhập, Google OAuth và khôi phục mật khẩu.
    \item Kiểm tra quyền truy cập theo người dùng và project.
    \item Lưu dữ liệu nghiệp vụ và trạng thái thực thi.
    \item Tạo test run và gửi yêu cầu sang agent-worker.
    \item Nhận callback step, evidence và kết quả cuối cùng.
\end{itemize}

Backend không trực tiếp điều khiển trình duyệt mà chỉ điều phối các yêu cầu thực thi sang agent-worker.

% ---------------------------------------------------------
\subsection{Luồng giao tiếp tổng quan}

Frontend gửi yêu cầu nghiệp vụ đến backend. Backend kiểm tra dữ liệu, lưu bản ghi cần thiết và chuyển yêu cầu thực thi sang agent-worker. Worker chạy test trên trình duyệt, ghi nhận step log, screenshot và evidence, sau đó callback kết quả để backend cập nhật test run và cung cấp dữ liệu cho giao diện.

% =========================================================
\section{Thiết kế dữ liệu và lưu trữ}
% =========================================================

Cơ sở dữ liệu được thiết kế xoay quanh vòng đời kiểm thử, từ tạo project và test case đến chạy kiểm thử, lưu replay script, evidence và kết quả phân tích.

Chương này chỉ trình bày các bảng cốt lõi và nhóm dữ liệu hỗ trợ. Toàn bộ schema vật lý, migration, chỉ mục và ràng buộc chi tiết không thuộc phạm vi trình bày chính.

Các nguyên tắc thiết kế dữ liệu gồm:

\begin{itemize}
    \item \textbf{Truy vết đầy đủ}: mỗi test run phải liên kết được với project, test case, replay script, dataset và evidence liên quan.
    
    \item \textbf{Tách dữ liệu thiết kế và dữ liệu thực thi}: test case mô tả nội dung cần kiểm thử; test run biểu diễn một lần thực thi cụ thể.
    
    \item \textbf{Hỗ trợ phiên bản}: lưu phiên bản test case để truy vết nội dung tại thời điểm chạy.
    
    \item \textbf{Hỗ trợ dữ liệu bán cấu trúc}: replay script, runtime config, action history và evidence summary có thể được lưu bằng JSON/JSONB.
    
    \item \textbf{Hỗ trợ phân tích lỗi}: step log và evidence được lưu riêng để xác định bước xảy ra lỗi.
\end{itemize}

\subsection{Luồng tạo project và scan/crawl website}

Người dùng khai báo tên project, mô tả, base URL và cấu hình liên quan. Backend kiểm tra dữ liệu, tạo project và xác định phạm vi website mục tiêu.

Khi được chạy với dataset, các biến trong script được thay thế bằng dữ liệu của từng dòng. Nếu locator, action hoặc assertion không còn phù hợp, hệ thống ghi nhận \texttt{failed} hoặc \texttt{error} và lưu evidence để phân tích.

\section{Thiết kế agent-worker và cơ chế thực thi}
% =========================================================

Agent-worker là thành phần trực tiếp khởi tạo trình duyệt, chạy agent hoặc replay script và ghi nhận evidence. Việc tách worker khỏi backend giúp backend tập trung vào nghiệp vụ, trong khi worker xử lý các tác vụ tiêu tốn nhiều tài nguyên.

\begin{figure}[H]
    \centering
    \includegraphics[
        width=\textwidth,
        height=0.78\textheight,
        keepaspectratio
    ]{images/chapter4/agent_worker_design.png}
    \caption{Thiết kế bên trong agent-worker}
    \label{fig:agent_worker_design}
\end{figure}

Trong chế độ agent, worker nhận goal, test plan, dữ liệu đầu vào và cấu hình runtime. Worker phối hợp LLM với thành phần tự động hóa trình duyệt để thực hiện action và kiểm tra trạng thái.

Trong chế độ replay, worker đọc replay script và phát lại các action theo thứ tự. Cách này giảm chi phí gọi LLM và tăng tính lặp lại của kết quả.

Agent-worker cần kiểm soát timeout, số phiên trình duyệt đồng thời, allowed domains và các lỗi như website không tải được, locator thất bại hoặc assertion không đạt.

% =========================================================
\section{Thiết kế cơ chế hỗ trợ bảo trì kiểm thử}
% =========================================================

\subsection{Object Repository và State Anchors}

Object Repository lưu thông tin nhận diện phần tử theo project. Mỗi object có thể gồm tên, page key, selector collection, attribute snapshot và nguồn tạo. Dữ liệu này hỗ trợ tái sử dụng selector và cung cấp thêm ngữ cảnh khi agent cần tìm phần tử.

State Anchors là các điều kiện xác nhận trạng thái quan trọng, chẳng hạn URL thay đổi, phần tử xuất hiện, input có giá trị đúng hoặc thông báo lỗi được hiển thị. Các anchor hoặc assertion được sử dụng để xác định verdict của test case.

Locator fallback là một nhánh trong luồng agent. Khi cách nhận diện ban đầu thất bại, agent có thể quan sát lại giao diện hoặc sử dụng selector và thuộc tính đã lưu để thử cách định vị khác. Cơ chế này không thay thế việc cập nhật test case khi luồng nghiệp vụ thay đổi lớn.

% ---------------------------------------------------------
\subsection{Quản lý test suite và dữ liệu kiểm thử}

Test suite gom nhiều test case thành một nhóm chạy. Mỗi test case trong suite có thể được cấu hình chạy bằng agent hoặc replay script; kết quả được tổng hợp theo từng lượt thực thi.

Dữ liệu kiểm thử được lưu thành dataset, trong đó mỗi dòng là một bộ input. Khi chạy replay script với dataset, hệ thống tạo kết quả riêng cho từng dòng dữ liệu.

Trước khi chạy, hệ thống kiểm tra cấu trúc dataset với các biến của replay script dựa trên số lượng trường và kiểu dữ liệu. Cấu hình không đầy đủ hoặc dataset không tương thích sẽ bị chặn.

\section{Kết chương}
