\section{Kiến trúc tổng thể hệ thống}

Từ các yêu cầu đã phân tích ở Chương~\ref{Chapter3}, hệ thống được thiết kế theo hướng tách biệt giữa frontend, backend API, cơ sở dữ liệu và agent-worker. Mục tiêu của kiến trúc là hỗ trợ quy trình kiểm thử tự động ứng dụng web, bao gồm tạo project, tạo test case từ mô tả tự nhiên, chạy test bằng agent, lưu evidence, sinh replay script và chạy lại kịch bản kiểm thử.

Các mục tiêu thiết kế chính gồm:

\begin{itemize}
    \item \textbf{Tách biệt trách nhiệm}: frontend xử lý giao diện, backend xử lý nghiệp vụ, agent-worker xử lý browser automation.
    \item \textbf{Hỗ trợ ngôn ngữ tự nhiên}: người dùng có thể mô tả mục tiêu kiểm thử bằng prompt hoặc goal.
    \item \textbf{Kết hợp agent và replay}: agent dùng cho lần chạy linh hoạt ban đầu, replay script dùng cho kiểm thử hồi quy.
    \item \textbf{Lưu vết kết quả}: mỗi lần chạy cần có trạng thái, verdict, log, screenshot, evidence và thông tin lỗi.
    \item \textbf{Hỗ trợ bảo trì}: hệ thống sử dụng object repository, locator fallback, state anchors và fast-forward inspect.
    \item \textbf{Kiểm soát chi phí}: giảm số lần gọi LLM khi đã có replay script ổn định.
\end{itemize}

\section{Thiết kế dữ liệu và lưu trữ}

Cơ sở dữ liệu được thiết kế để phục vụ toàn bộ vòng đời kiểm thử tự động, từ tạo project, scan website, tạo test case, chạy test, lưu evidence, sinh replay script đến phân tích lỗi. PostgreSQL 16 được sử dụng làm cơ sở dữ liệu chính.

Các nguyên tắc thiết kế dữ liệu gồm:

\begin{itemize}
    \item \textbf{Truy vết đầy đủ}: mỗi test run cần liên kết được với project, test case, script, evidence và người kích hoạt.
    \item \textbf{Tách dữ liệu theo nghiệp vụ}: dữ liệu được chia theo project/config, test case/script, execution/evidence, dataset/test suite và object repository.
    \item \textbf{Hỗ trợ dữ liệu bán cấu trúc}: replay script, selector collection, evidence summary và runtime config được lưu dạng JSON/JSONB.
    \item \textbf{Hỗ trợ chạy lại}: hệ thống lưu script, params, evidence và lịch sử để phục vụ kiểm thử hồi quy.
    \item \textbf{Giảm phụ thuộc vào selector cứng}: object repository lưu nhiều selector để hỗ trợ locator fallback.
\end{itemize}

Thay vì trình bày toàn bộ database trong một ERD lớn, mô hình dữ liệu được chia thành bốn nhóm chính.

\subsection{Luồng scan/crawl website và sinh test case}

Luồng sinh test case bắt đầu từ prompt hoặc goal do người dùng nhập vào. Backend kết hợp prompt với thông tin project hoặc scan result nếu có, sau đó gọi AI để sinh test case gồm title, goal và test plan.

\section{Thiết kế cơ chế hỗ trợ bảo trì kiểm thử}

\subsection{Object Repository, State Anchors và Locator Fallback}

Object Repository lưu thông tin định vị phần tử giao diện theo hướng có thể tái sử dụng. Mỗi object có thể chứa tên object, page key, mô tả, selector collection, attribute snapshot và nguồn tạo object. Khi replay script cần thao tác với phần tử, hệ thống có thể tham chiếu đến object thay vì phụ thuộc hoàn toàn vào một selector cứng.

State Anchors là các điều kiện xác nhận trạng thái quan trọng của giao diện, chẳng hạn như URL thay đổi, văn bản xuất hiện, phần tử hiển thị hoặc thông báo lỗi xuất hiện. Nếu action chạy được nhưng anchor không đạt, test run được ghi nhận là failed.

Locator fallback trong hệ thống chỉ hỗ trợ phục hồi trước một số thay đổi nhỏ của giao diện. Khi chạy replay, hệ thống không gọi LLM fallback mà thử locator chính trước, sau đó thử các locator dự phòng trong selector collection.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{images/chapter4/self_healing_flow.png}
\caption{Cơ chế locator fallback khi locator chính thất bại}
\label{fig:locator_fallback_flow}
\end{figure}

Nếu locator dự phòng tìm được phần tử phù hợp, worker tiếp tục thực hiện action và ghi nhận quá trình fallback. Nếu tất cả locator đều thất bại, hệ thống lưu lỗi và evidence để người dùng có cơ sở sửa test case hoặc cập nhật object repository.

\subsection{Quản lý test suite và dữ liệu kiểm thử}

Test suite giúp gom nhóm nhiều test case để chạy theo lô, phù hợp với kiểm thử hồi quy. Dữ liệu kiểm thử giúp chạy cùng một test case với nhiều bộ input khác nhau, ví dụ tài khoản hợp lệ, sai mật khẩu, thiếu username hoặc dữ liệu sai định dạng.

\section{Kết chương}

Chương này đã trình bày kiến trúc và thiết kế hệ thống kiểm thử tự động ứng dụng web. Hệ thống được thiết kế theo hướng tách biệt giữa frontend, backend API, cơ sở dữ liệu và agent-worker, giúp phân tách rõ giao diện, nghiệp vụ, lưu trữ và thực thi kiểm thử.

Chương cũng trình bày thiết kế dữ liệu, các luồng xử lý cốt lõi, cơ chế hỗ trợ bảo trì, bảo mật và các trade-off quan trọng. Đây là cơ sở để trình bày quá trình hiện thực và tích hợp hệ thống ở chương tiếp theo.
