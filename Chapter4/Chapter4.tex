\section{Kiến trúc tổng thể hệ thống}

Từ các yêu cầu đã phân tích ở Chương~\ref{Chapter3}, hệ thống được thiết kế theo hướng tách biệt giữa giao diện người dùng, backend quản lý nghiệp vụ, cơ sở dữ liệu và agent-worker thực thi kiểm thử trên trình duyệt. Mục tiêu của kiến trúc là hỗ trợ quy trình kiểm thử tự động ứng dụng web, bao gồm tạo project, tạo test case từ mô tả tự nhiên, chạy test bằng agent, lưu evidence, sinh replay script và chạy lại kịch bản kiểm thử.

Các mục tiêu thiết kế chính gồm:

\begin{itemize}
    \item \textbf{Tách biệt trách nhiệm}: frontend xử lý giao diện, backend xử lý nghiệp vụ và điều phối, agent-worker xử lý browser automation.
    \item \textbf{Hỗ trợ kiểm thử bằng ngôn ngữ tự nhiên}: người dùng có thể mô tả mục tiêu kiểm thử bằng prompt hoặc goal.
    \item \textbf{Kết hợp agent và replay script}: agent dùng cho lần chạy linh hoạt ban đầu, replay script dùng cho các lần kiểm thử hồi quy.
    \item \textbf{Lưu vết đầy đủ}: mỗi lần chạy cần có trạng thái, verdict, log, screenshot, evidence và thông tin lỗi.
    \item \textbf{Tăng khả năng bảo trì}: hệ thống hỗ trợ object repository, state anchors và self-healing ở mức locator fallback trong quá trình agent thực thi.
    \item \textbf{Kiểm soát chi phí và tài nguyên}: giảm số lần gọi LLM khi đã có replay script ổn định và giới hạn số phiên trình duyệt chạy đồng thời.
\end{itemize}

\subsection{Kiến trúc component cấp hệ thống}

Kiến trúc tổng thể của hệ thống gồm các thành phần chính: frontend, backend API, PostgreSQL database, agent-worker, Browser-use, Playwright, LLM và website mục tiêu. Các thành phần phối hợp với nhau thông qua API và callback để hoàn thành quy trình kiểm thử.

Backend kiểm tra quyền truy cập, lưu dữ liệu nghiệp vụ vào PostgreSQL và gửi yêu cầu thực thi sang agent-worker. Agent-worker sử dụng Browser-use, Playwright và LLM để chạy test trên trình duyệt, sau đó callback kết quả về backend.

\subsection{Luồng giao tiếp tổng quan}

Về tổng thể, frontend gửi yêu cầu nghiệp vụ đến backend. Backend lưu dữ liệu vào database và gửi yêu cầu chạy test sang worker khi cần thực thi. Worker chạy test trên trình duyệt, ghi nhận log, screenshot, action history và callback kết quả về backend. Backend cập nhật test run, evidence summary, replay script nếu có và trả dữ liệu để frontend hiển thị báo cáo.

\section{Thiết kế dữ liệu và lưu trữ}

Cơ sở dữ liệu được thiết kế để phục vụ vòng đời kiểm thử tự động, từ tạo project, tạo test case, chạy test, lưu evidence, sinh replay script đến phân tích lỗi. PostgreSQL 16 được sử dụng làm cơ sở dữ liệu chính.

Trong chương này, phần thiết kế dữ liệu không trình bày toàn bộ schema vật lý của hệ thống. Thay vào đó, nội dung tập trung vào các bảng cốt lõi giúp người đọc hiểu dữ liệu chính được tổ chức và liên kết như thế nào trong vòng đời kiểm thử. Các bảng phụ trợ như cấu hình runtime, phiên đăng nhập, dữ liệu phân loại, lịch sử sinh dữ liệu bằng AI, test suite hoặc các bảng phục vụ vận hành được mô tả ở mức tổng quan và sẽ được trình bày chi tiết hơn ở Chương~\ref{Chapter5} nếu cần.

Các nguyên tắc thiết kế dữ liệu gồm:

\begin{itemize}
    \item \textbf{Truy vết đầy đủ}: mỗi test run cần liên kết được với project, test case, replay script, evidence và người kích hoạt.
    \item \textbf{Tách dữ liệu theo vòng đời kiểm thử}: dữ liệu được tổ chức xoay quanh project, test case, replay script, test run, step log, evidence và object repository.
    \item \textbf{Hỗ trợ dữ liệu bán cấu trúc}: replay script, action history, evidence summary, runtime config hoặc thông tin hỗ trợ định vị phần tử có thể được lưu dạng JSON/JSONB để dễ mở rộng.
    \item \textbf{Hỗ trợ kiểm thử hồi quy}: hệ thống lưu lịch sử chạy, replay script và evidence để phục vụ các lần kiểm thử sau.
    \item \textbf{Hỗ trợ phân tích lỗi}: test run, step log và evidence được tách riêng để người dùng có thể xem lại quá trình thực thi.
\end{itemize}

\section{Thiết kế các luồng xử lý cốt lõi}

Phần này mô tả các luồng xử lý chính của hệ thống. Các luồng được trình bày theo vòng đời kiểm thử: chuẩn bị project, tạo test case, chạy kiểm thử bằng agent, sinh replay script, chạy replay và ghi nhận kết quả.

\subsection{Luồng tạo project, scan/crawl website và sinh test case}

Luồng scan/crawl website được dùng để xây dựng thông tin nền ban đầu về website trong phạm vi project, bao gồm các URL có thể truy cập, sitemap và interaction map. Chức năng này không phân tích sâu toàn bộ logic giao diện mà chỉ cung cấp ngữ cảnh ban đầu cho các bước tạo test case và chạy agent.

Luồng sinh test case bắt đầu từ prompt hoặc goal do người dùng nhập vào. Backend kết hợp prompt với thông tin project hoặc scan result nếu có, sau đó gọi AI để sinh test case gồm title, goal và test plan. Test case được lưu để người dùng xem lại, chỉnh sửa hoặc chạy thử.

\subsection{Luồng chạy bằng goal-based agent}

Goal-based agent là chế độ chạy linh hoạt nhất của hệ thống. Agent đọc mục tiêu kiểm thử, quan sát giao diện hiện tại và quyết định hành động tiếp theo dựa trên trạng thái thực tế của website. Khi agent gặp khó khăn trong việc nhận diện phần tử giao diện, cơ chế locator fallback được xử lý như một nhánh trong chính luồng agent execution, thay vì tách thành một luồng riêng.

Trong luồng này, người dùng chọn test case và yêu cầu chạy bằng agent. Backend tạo test run và gửi yêu cầu sang worker. Worker sử dụng Browser-use, Playwright và LLM để quan sát giao diện, suy luận hành động và thao tác trên website mục tiêu. Nếu locator hoặc cách nhận diện phần tử ban đầu không còn phù hợp, agent có thể quan sát lại giao diện, sử dụng ngữ cảnh hoặc thông tin phần tử có sẵn để thử cách nhận diện khác và tiếp tục thực thi. Trong quá trình chạy, hệ thống ghi nhận log, screenshot, action history và evidence. Khi test kết thúc, worker trả kết quả về backend. Nếu phiên chạy thành công, hệ thống có thể sinh replay script từ lịch sử hành động để phục vụ các lần chạy lại sau.

Nếu locator, action hoặc assertion trong script không còn phù hợp, test run được ghi nhận là failed hoặc error, đồng thời hệ thống lưu evidence để người dùng phân tích và cập nhật lại test case, object repository hoặc replay script khi cần.

\section{Thiết kế agent-worker và cơ chế thực thi}

Agent-worker là thành phần kỹ thuật lõi của hệ thống vì đây là nơi trực tiếp khởi tạo trình duyệt, điều phối Browser-use agent, chạy replay script bằng Playwright và ghi nhận evidence. Việc tách agent-worker khỏi backend giúp backend tập trung vào nghiệp vụ và API, trong khi worker xử lý các tác vụ nặng hơn như browser automation, gọi LLM và lưu bằng chứng thực thi.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{images/chapter4/agent_worker_design.png}
\caption{Thiết kế bên trong agent-worker}
\label{fig:agent_worker_design}
\end{figure}

Khi chạy bằng goal-based agent, worker nhận test case và runtime config từ backend, sau đó khởi tạo browser context trong phạm vi website được phép truy cập. Worker sử dụng Browser-use agent để quan sát giao diện và gọi LLM nhằm suy luận hành động tiếp theo. Các thao tác trên trình duyệt được thực hiện thông qua Playwright. Trong quá trình này, worker ghi nhận action history, log, screenshot và các evidence liên quan.

Khi chạy bằng replay script, worker không cần gọi LLM ở từng bước mà đọc script JSON đã lưu và dùng Playwright để phát lại các action theo thứ tự. Cách này phù hợp với kiểm thử hồi quy vì giảm chi phí gọi LLM và giúp kết quả ổn định hơn so với chế độ agent.

Agent-worker cũng cần có các cơ chế kiểm soát vận hành như timeout, giới hạn số phiên trình duyệt chạy đồng thời, allowed domains và xử lý lỗi khi website không tải được, locator thất bại hoặc assertion không đạt. Các lỗi này được ghi nhận vào test run và evidence để người dùng có thể phân tích sau khi chạy.

\section{Thiết kế cơ chế hỗ trợ bảo trì kiểm thử}

\subsection{Object Repository và State Anchors}

Object Repository được dùng để lưu thông tin định vị phần tử giao diện theo hướng có thể tái sử dụng. Mỗi object có thể chứa tên object, page key, mô tả, selector collection, attribute snapshot và nguồn tạo object. Thông tin này giúp hệ thống có thêm dữ liệu tham chiếu khi agent cần nhận diện phần tử giao diện trong quá trình thực thi, đồng thời hỗ trợ tái sử dụng thông tin phần tử cho các lần chạy sau.

State Anchors là các điều kiện xác nhận trạng thái quan trọng của giao diện, chẳng hạn như URL thay đổi, văn bản xuất hiện, phần tử hiển thị, giá trị input đúng hoặc thông báo lỗi xuất hiện. Khi chạy test, hệ thống không chỉ thực hiện action mà còn cần kiểm tra các anchor/assertion này để xác định test case đạt hay không đạt.

Self-healing ở mức locator fallback không được tách thành một quy trình độc lập trong thiết kế, mà được xem là một phần của luồng chạy bằng agent. Khi agent không nhận diện được phần tử theo cách ban đầu, hệ thống có thể sử dụng lại ngữ cảnh giao diện, thuộc tính phần tử hoặc thông tin trong object repository để hỗ trợ agent thử cách nhận diện khác. Cơ chế này chỉ hỗ trợ một số thay đổi nhỏ của giao diện và không thay thế việc cập nhật lại test case khi luồng nghiệp vụ thay đổi lớn.

\subsection{Quản lý test suite và dữ liệu kiểm thử}

Test suite giúp gom nhóm nhiều test case theo một mục tiêu kiểm thử lớn hơn, ví dụ nhóm kiểm thử đăng nhập, quản lý dữ liệu hoặc kiểm thử hồi quy cho một module. Khi chạy test suite, hệ thống lần lượt chạy các test case trong danh sách và tổng hợp kết quả pass, failed hoặc error.

Dữ liệu kiểm thử giúp chạy cùng một test case với nhiều bộ input khác nhau. Ví dụ, một test case đăng nhập có thể được chạy với tài khoản hợp lệ, sai mật khẩu, thiếu username hoặc dữ liệu sai định dạng. Cách thiết kế này giúp người dùng không cần tạo nhiều test case gần giống nhau chỉ để thay đổi dữ liệu đầu vào.

\section{Kết chương}

Chương này đã trình bày kiến trúc và thiết kế hệ thống kiểm thử tự động ứng dụng web. Hệ thống được thiết kế theo hướng tách biệt giữa frontend, backend API, cơ sở dữ liệu và agent-worker, giúp phân tách rõ phần giao diện, nghiệp vụ, lưu trữ và thực thi kiểm thử.

Chương cũng trình bày thiết kế dữ liệu ở mức cốt lõi, các luồng xử lý chính, thiết kế agent-worker, cơ chế hỗ trợ bảo trì kiểm thử, nguyên tắc bảo mật và các đánh đổi quan trọng. Đối với thiết kế dữ liệu, chương chỉ tập trung vào các bảng cốt lõi phục vụ vòng đời kiểm thử, đồng thời mô tả tổng quan các nhóm bảng hỗ trợ để tránh làm mô hình dữ liệu quá phức tạp. Các chi tiết hiện thực cụ thể của schema vật lý và các bảng phụ trợ sẽ được trình bày ở Chương~\ref{Chapter5}.

Đặc biệt, chương đã làm rõ vai trò của agent-worker trong việc điều phối Browser-use, Playwright và LLM để thực thi kiểm thử, đồng thời phân biệt giữa chế độ chạy linh hoạt bằng agent và chế độ chạy ổn định hơn bằng replay script. Đây là cơ sở để trình bày quá trình hiện thực và tích hợp hệ thống ở chương tiếp theo.
