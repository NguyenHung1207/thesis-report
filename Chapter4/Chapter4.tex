\section{Kiến trúc tổng thể hệ thống}

Từ các yêu cầu đã phân tích ở Chương~\ref{Chapter3}, hệ thống được thiết kế theo hướng tách biệt giữa giao diện người dùng, backend quản lý nghiệp vụ, cơ sở dữ liệu và agent-worker thực thi kiểm thử trên trình duyệt. Mục tiêu của kiến trúc là hỗ trợ toàn bộ quy trình kiểm thử tự động ứng dụng Web, bao gồm tạo project, tạo test case từ mô tả tự nhiên, chạy test bằng agent, lưu evidence, sinh replay script và chạy lại kịch bản kiểm thử với nhiều bộ dữ liệu.

Các mục tiêu thiết kế chính gồm:

\begin{itemize}
    \item \textbf{Tách biệt trách nhiệm}: frontend xử lý giao diện, backend xử lý nghiệp vụ và điều phối, agent-worker xử lý browser automation.
    \item \textbf{Hỗ trợ kiểm thử bằng ngôn ngữ tự nhiên}: người dùng có thể mô tả mục tiêu kiểm thử bằng prompt hoặc goal.
    \item \textbf{Kết hợp agent và replay script}: agent dùng cho lần chạy linh hoạt ban đầu, replay script dùng cho các lần kiểm thử hồi quy.
    \item \textbf{Lưu vết đầy đủ}: mỗi lần chạy cần có trạng thái, verdict, log, screenshot, evidence và thông tin lỗi.
    \item \textbf{Tăng khả năng bảo trì}: hệ thống hỗ trợ object repository, locator fallback, self-healing và fast-forward inspect.
    \item \textbf{Kiểm soát chi phí và tài nguyên}: giảm số lần gọi LLM khi đã có replay script ổn định, đồng thời giới hạn số phiên trình duyệt chạy đồng thời.
\end{itemize}

\section{Thiết kế dữ liệu và lưu trữ}

Cơ sở dữ liệu được thiết kế để phục vụ toàn bộ vòng đời kiểm thử tự động, từ tạo project, scan website, tạo test case, chạy test, lưu evidence, sinh replay script đến phân tích lỗi. PostgreSQL 16 được sử dụng làm cơ sở dữ liệu chính.

Các nguyên tắc thiết kế dữ liệu gồm:

\begin{itemize}
    \item \textbf{Truy vết đầy đủ}: mỗi test run cần liên kết được với project, test case, script, evidence và người kích hoạt.
    \item \textbf{Tách dữ liệu theo nhóm nghiệp vụ}: dữ liệu được chia theo nhóm project/config, test case/script, execution/evidence, dataset/test suite và object repository.
    \item \textbf{Hỗ trợ dữ liệu bán cấu trúc}: replay script, selector collection, evidence summary và runtime config được lưu dạng JSON/JSONB.
    \item \textbf{Hỗ trợ chạy lại}: hệ thống lưu script, params, evidence và lịch sử để phục vụ các lần kiểm thử sau.
    \item \textbf{Giảm phụ thuộc vào selector cứng}: object repository lưu nhiều selector/attribute để hỗ trợ locator fallback.
\end{itemize}

Thay vì trình bày toàn bộ database trong một ERD lớn, mô hình dữ liệu được chia thành bốn nhóm chính để dễ theo dõi.

\section{Thiết kế các luồng xử lý cốt lõi}

Phần này mô tả các luồng xử lý chính của hệ thống. Các luồng chi tiết được thể hiện bằng hình để tránh trình bày quá dài bằng văn bản.

\subsection{Luồng scan/crawl website và sinh test case}

Luồng scan/crawl website được dùng để xây dựng thông tin nền ban đầu về website trong phạm vi project, bao gồm các URL có thể truy cập, sitemap và interaction map. Chức năng này không phân tích sâu toàn bộ logic giao diện mà chỉ cung cấp ngữ cảnh ban đầu cho các bước tạo test case và chạy agent.

Luồng sinh test case bắt đầu từ prompt hoặc goal do người dùng nhập vào. Backend kết hợp prompt với thông tin project hoặc scan result nếu có, sau đó gọi AI để sinh test case gồm title, goal và test plan. Test case được lưu ở trạng thái phù hợp để người dùng xem lại, chỉnh sửa hoặc chạy thử.

\subsection{Luồng chạy bằng goal-based agent}

Goal-based agent là chế độ chạy linh hoạt nhất của hệ thống. Agent đọc mục tiêu kiểm thử, quan sát giao diện hiện tại và quyết định hành động tiếp theo dựa trên trạng thái thực tế của website.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{images/chapter4/flow_goal_based_agent.png}
\caption{Luồng chạy test bằng goal-based agent}
\label{fig:flow_goal_based_agent}
\end{figure}

Trong luồng này, người dùng chọn test case và yêu cầu chạy bằng agent. Backend tạo test run, chuẩn bị payload gồm goal, prompt, input params, runtime config, browser profile và callback URL rồi gửi sang worker. Worker khởi tạo trình duyệt bằng Playwright, tạo Browser-use agent, gọi LLM để suy luận hành động và ghi nhận log, screenshot, action history trong quá trình chạy. Khi test kết thúc, worker callback kết quả về backend. Nếu chạy thành công, hệ thống sinh replay script từ lịch sử hành động.

\subsection{Luồng chạy bằng replay script}

Replay script được sử dụng khi test case đã có kịch bản phát lại. Đây là chế độ phù hợp cho kiểm thử hồi quy vì không cần agent suy luận lại toàn bộ luồng kiểm thử.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{images/chapter4/flow_replay_script.png}
\caption{Luồng chạy test bằng replay script}
\label{fig:flow_replay_script}
\end{figure}

Trong chế độ replay, worker đọc script JSON đã lưu và sử dụng Playwright để chạy từng step theo thứ tự. Mỗi step có thể là navigate, fill, click hoặc assertion. Nếu step tham chiếu object repository, worker lấy selector collection của object, thử locator chính trước và thử locator dự phòng nếu locator chính thất bại. Chế độ replay không sử dụng LLM fallback; nếu locator và assertion đều không đạt, test run được ghi nhận là fail hoặc error.

\subsection{Luồng fast-forward inspect}

Fast-forward inspect là cơ chế hỗ trợ sửa step trong replay script. Khi một step bị lỗi, worker chạy nhanh các bước trước target step để đưa trình duyệt đến trạng thái ngay trước bước cần kiểm tra.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{images/chapter4/flow_fast_forward_inspect.png}
\caption{Luồng fast-forward inspect để hỗ trợ sửa step}
\label{fig:flow_fast_forward_inspect}
\end{figure}

Sau khi đến đúng trạng thái, worker trả về screenshot, URL hiện tại và danh sách phần tử tương tác trên trang. Người dùng có thể xem trạng thái giao diện, nhập hướng dẫn sửa step hoặc dùng AI hỗ trợ đề xuất cập nhật. Cơ chế này giúp rút ngắn thời gian sửa test case, đặc biệt với các luồng dài như đăng nhập hoặc thao tác qua nhiều màn hình.

\subsection{Luồng sinh report và evidence}

Trong quá trình chạy test, worker ghi nhận log, screenshot, action history và kết quả từng bước. Khi test kết thúc, worker tổng hợp verdict, error message và evidence summary rồi callback về backend. Backend cập nhật test run, lưu evidence và cung cấp dữ liệu để frontend hiển thị report.

Report cần thể hiện rõ test pass/fail, lỗi xảy ra ở bước nào và evidence liên quan. Với người dùng kỹ thuật, hệ thống có thể hiển thị thêm log chi tiết, action history, assertion failure hoặc thông tin locator để phục vụ quá trình sửa lỗi.

\section{Thiết kế cơ chế hỗ trợ bảo trì kiểm thử}

\subsection{Object Repository, State Anchors và Self-healing}

Object Repository được dùng để lưu thông tin định vị phần tử giao diện theo hướng có thể tái sử dụng. Mỗi object có thể chứa tên object, page key, mô tả, selector collection, attribute snapshot và nguồn tạo object. Khi replay script cần thao tác với một phần tử, hệ thống có thể tham chiếu đến object thay vì phụ thuộc hoàn toàn vào một selector cứng.

State Anchors là các điều kiện xác nhận trạng thái quan trọng của giao diện, chẳng hạn như URL thay đổi, văn bản xuất hiện, phần tử hiển thị, giá trị input đúng hoặc thông báo lỗi xuất hiện. Khi replay script chạy, worker không chỉ thực hiện action mà còn kiểm tra các anchor/assertion này. Nếu action chạy được nhưng anchor không đạt, test run được ghi nhận là fail.

Self-healing trong hệ thống được hiểu là khả năng phục hồi trước một số thay đổi nhỏ của giao diện, không phải khả năng tự sửa mọi lỗi kiểm thử. Khi chạy bằng agent, hệ thống có thể quan sát lại giao diện và suy luận hành động phù hợp. Khi chạy replay, hệ thống không gọi LLM fallback mà thử locator chính trước, sau đó thử các locator dự phòng trong selector collection.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{images/chapter4/self_healing_flow.png}
\caption{Cơ chế self-healing khi locator chính thất bại}
\label{fig:self_healing_flow}
\end{figure}

Nếu locator dự phòng tìm được phần tử phù hợp, worker tiếp tục thực hiện action và ghi nhận quá trình fallback. Nếu tất cả locator đều thất bại, hệ thống lưu lỗi và evidence để người dùng có cơ sở sửa test case hoặc cập nhật object repository.

\subsection{Quản lý test suite và dữ liệu kiểm thử}

Test suite giúp gom nhóm nhiều test case theo một mục tiêu kiểm thử lớn hơn, ví dụ nhóm kiểm thử đăng nhập, quản lý dữ liệu hoặc kiểm thử hồi quy cho một module. Khi chạy test suite, hệ thống lần lượt chạy các test case trong danh sách và tổng hợp kết quả pass, fail hoặc error.

Dữ liệu kiểm thử giúp chạy cùng một test case với nhiều bộ input khác nhau. Ví dụ, một test case đăng nhập có thể được chạy với tài khoản hợp lệ, sai mật khẩu, thiếu username hoặc dữ liệu sai định dạng. Cách thiết kế này giúp người dùng không cần tạo nhiều test case gần giống nhau chỉ để thay đổi dữ liệu đầu vào.

\section{Kết chương}

Chương này đã trình bày kiến trúc và thiết kế hệ thống kiểm thử tự động ứng dụng Web. Hệ thống được thiết kế theo hướng tách biệt giữa frontend, backend API, PostgreSQL database và agent-worker. Cách tách này giúp backend tập trung vào quản lý nghiệp vụ, trong khi agent-worker đảm nhiệm các tác vụ tiêu tốn tài nguyên như chạy trình duyệt, gọi Browser-use agent, thực thi replay script và ghi nhận evidence.

Chương cũng đã trình bày thiết kế dữ liệu theo từng nhóm chức năng, các luồng xử lý cốt lõi, cơ chế hỗ trợ bảo trì kiểm thử, nguyên tắc bảo mật và các trade-off quan trọng trong thiết kế. Đây là cơ sở để trình bày chi tiết hơn về quá trình hiện thực hệ thống và đánh giá kết quả ở các chương tiếp theo.
